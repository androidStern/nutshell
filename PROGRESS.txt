Setup/onboarding concern to preserve for next agent:

- Current `nutshell setup` asks which plugins to enable, loops selected plugins through their `setup.summarize/run/verify`, optionally offers provider archive imports, commits plugin enabled/setup state, then handles app-level Full Disk Access/background-agent enablement.
- Built-in YouTube, Twitter/X, Apple Notes, and Podcasts setup callbacks currently defer real browser/protected-data/automation checks. They mostly show text and return no findings, so selected plugins can be marked `ready` before login/permission has actually been proven.
- Browser login failures currently appear later through `doctor`, `health`, or `sync`; `setup` does not wait for Google/X login and does not auto-resume after the user logs into Chrome.
- User concern: this is not a normal setup CLI experience. At each source step the user should be able to understand current state, skip, resume later, and find the right return/import commands without hidden knowledge.
- User concern: plugin-specific onboarding must remain plugin-owned. The setup orchestrator should not absorb YouTube/X/Notes/Podcasts-specific behavior in violation of the plugin design boundary.
- User requested a neutral handoff only; do not treat these notes as a prescribed redesign.
